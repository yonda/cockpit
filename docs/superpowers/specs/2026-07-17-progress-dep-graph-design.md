# progress レンズ: run 内を依存グラフ + ステージレールにする

- Issue: #172
- 日付: 2026-07-17
- 対象: `app/_components/ProgressLens.tsx` とその導出ロジック

## 背景

progress レンズは「平常時に端末を覗かずに、どの issue がどう分解され今どうなっているか」を見るための画面である。現状は run 内のノードを `<ol>` の縦一列で描き、依存の深さを `marginLeft: depth * 16` のインデントだけで表現している。

この表現では run 内で 2 つのことが読めない。

1. **依存の形** — 「このノードは誰待ちか」がインデント量でしか出ていない。並び順は進捗ファイルの記載順のままでトポロジカル順ではないため、依存元と依存先が離れた行にあると連なりを目で追えない。
2. **各ノードの現在地** — `liveStatus` と PR/issue バッジが同じ行に並列に並ぶだけで、「queued → implementing → reviewing → merged のどこにいるか」が読み取りづらい。

## 設計

run 内を**左→右の依存グラフ**として描き、各ノードの箱に**ステージレール**を持たせる。

```
depth 0                        depth 1

┌─────────────────────┐
│ t48 (対象外確認)     │
│ ●━●─○─○ implementing │
│ issue #48 · open     │
└─────────────────────┘

┌─────────────────────┐        ┌─────────────────────┐
│ t42 (BE)             │───────▶│ t43 (FE)             │
│ ●─○─○─○ queued       │        │ ●─○─○─○ blocked@queued│
│ issue #42 · open     │        │ issue #43 · open     │
└─────────────────────┘        └─────────────────────┘
```

- **列 = 依存の深さ**（トポロジカルなランク）
- **線 = 依存**。線が「誰待ちか」を描くため、「◯◯待ち」という文字表現は持たせない
- **ノードの箱** = タイトル / 4点レール + 現在地ラベル / `activity` / GitHub バッジ

`activity`（進捗ファイルが持つ人間が読む一行。例: 「実装中: xxx を追加」）は現行レンズが表示しており、機械的な状態表現では代替できない唯一の自然言語の signal なので箱に残す。長い場合は 1 行で省略する。

依存ゼロ・1ノードの run もグラフで描く。実装に分岐を作らず、run ごとに見た目が変わらないことを優先する。

### レイアウトはサーバ側で確定させる

グラフライブラリ (dagre / elkjs) もクライアント JS も導入しない。列幅・行高を固定値にすれば、依存の深さと列内の行番号から全ノードの座標がサーバ側で決まる。SVG の線もその座標から描けるため、`ProgressLens` はサーバコンポーネントのままでよい。

この判断は「グラフは浅い」という実データの性質に依存している。実データで最も依存があった run でも深さ 2・辺 2 本であり、交差最小化アルゴリズムを要する規模ではない。深い DAG が現れて破綻したら、その時点でライブラリ導入を再検討する。

## 導出ロジック

`ProgressLens` から独立した純関数に切り出し、ユニットテストを書く。`JoinedProgressFile` を入力とし、座標付きのグラフを返す。

### 1. ステージと条件の分解

現在の `LiveStatus` は段階 (`queued` / `implementing` / `reviewing`)、条件 (`blocked`)、PR 依存 (`handed_off`) が 1 つの enum に同居している。レールを描くにはこれを分解する必要がある。

**進捗ファイルのスキーマ (`lib/runs/progress.ts`) は変更しない。** cockpit 側の導出だけで解くため、issue-driver skill 側は無変更となる。「1事実=1つの持ち主」という観測契約を守り、GitHub 権威の事実 (PR のマージ状態) は join 層から重ねる。

導出結果の型:

```ts
type NodeStage = "queued" | "implementing" | "review" | "merged";
type NodeCondition = "normal" | "blocked" | "ok";
```

**stage は「GitHub の確定事実 > 進捗ファイルの自己申告 > 既定」の優先順で導出する。** 上から順に評価し、最初に当たったものを採用する。

| 順 | 条件 | stage | 根拠 |
|---|---|---|---|
| 1 | PR が `MERGED` | merged | GitHub 権威の事実 |
| 2 | PR が存在する（open / closed） | review | GitHub 権威の事実（PR がある = 実装は出た） |
| 3 | `liveStatus` が `reviewing` / `handed_off` | review | 進捗ファイルの自己申告 |
| 4 | `liveStatus` が `implementing` | implementing | 進捗ファイルの自己申告 |
| 5 | 上記いずれでもない | queued | 進んだ証拠がない |

condition は stage と直交に決める。

| 条件 | condition |
|---|---|
| PR が `MERGED` | ok |
| `liveStatus` が `blocked`、または PR が `CONFLICTING` | blocked |
| それ以外 | normal |

`blocked` は段階を持たない条件なので、表示は「条件＠段階」= `blocked @ queued` とする。規則 5 の `queued` は「進んだ証拠がない」の意であり、「まだ着手していない」と断定しているわけではない点に注意する。

**自己申告を信じる範囲について。** 当初は「証明できる最高段階」だけを採り、PR 番号を持たない `reviewing` / `handed_off` は `queued` に落とす設計だった（進捗ファイル側の記述漏れが可視化される、という理屈）。実データで描いて棄却した。

- 実際の run に `liveStatus: "handed_off"` かつ `prNumber: null` のノードがあり、`activity` は「完了。受入条件全パス」だった。これが**グレーの queued** として描かれた。不備の可視化にはなっておらず、未着手と見分けのつかない、もっともらしい嘘になっていた
- `implementing` の自己申告は信じるのに `reviewing` / `handed_off` の自己申告は信じない、という非対称に根拠がなかった

よって自己申告は段階によらず一律に信じ、GitHub の事実があればそれで上書きする。`blocked` だけは自己申告そのものが段階を持たないため、PR が無ければ `queued` に落ちる。

これにより「依存ゼロなのに blocked」なノードが **線が無いのに赤い** として絵で区別できる。依存待ちの blocked には線があるため、外部要因の blocked と視覚的に分離される。

### 2. 依存レイヤリング

- **列** — 各ノードの深さは既存の `nodeDepth` と同じ規則（`dependsOn` を辿った最大深さ）で決める。循環は既存実装と同様に経路集合で防ぎ、循環したノードは深さ 0 に落とす
- **行** — 列 0 は進捗ファイルの記載順。列 N+1 以降は「親（`dependsOn` 先）の行番号」を第一キー、記載順を第二キーにして並べる。これで親と子が近い行に来て線が短くなる
- 同じ行が競合した場合は順に下へ詰める

交差最小化は行わない。実データの規模では必要なく、必要になった時点で見直す。

### 3. 座標

固定値: 箱幅 236px / 箱高 88px / 列間 64px / 行間 8px。

```
x = col * (236 + 64)
y = row * (88 + 8)
```

辺は親の箱の右辺中央から子の箱の左辺中央への直線。親と子の行が異なる場合も直線で結ぶ（実データの規模では直交ルーティングは不要）。

キャンバスは `overflow-x: auto` でスクロールさせる。

## エラー処理・退行しないもの

- **既存の fail-safe を維持** — `githubFetchError` がある run はグラフを描いた上でエラー行を出す。ただし join に失敗すると `runJoin` は全ノードの `githubPullRequest` を null に潰すため、stage は自己申告だけで決まる（マージ済みのノードが `review` に見える等、実際より低く出る）。run 単位のエラー行だけではレール自体が退行していることが伝わらないので、**ノードのラベルに `· unverified` を付けて確定していないことを表示に出す**
- **escalation の表示は現状維持** — run レベル・ノードレベルとも既存の `EscalationNote` を残す。ノードの escalation は箱の高さを固定するためグラフの下に出し、どのノードのものか分かるようタイトルを添える。escalation を持つノードの箱は枠を alert 色にして対応づける
- 完了 (done) run の折りたたみ (#164)、`phase:monitoring` の扱い (#166) は現状維持
- 破損ファイルのスキップ表示は現状維持

### run ファイル名の契約を検査する

`list.ts` が `.json` で終わるファイルを無条件に run として読んでいたため、run ディレクトリに置かれた別物まで拾っていた。実データで、別の lead が置いたバックアップ (`41-dd-baseline-backup.json`) が `41.json` と同じ `repo#issueNumber` の run として読まれ、同じ React key で**同一の run が二重に描画**されていた。無関係な作業ファイルも「破損」としてスキップ件数に計上され、本物の破損を隠していた。

`<issueNumber>.json`（`/^\d+\.json$/`）だけを run として扱う。契約に合わないファイルは「破損」ではなく、そもそも run ではないので黙って無視する（`.DS_Store` と同じ扱い）。原子的書き込みの一時ファイル (`70.json.tmp`) が読まれないことも、これで担保される。

グラフ化とは別問題だが、この画面の表示が実際に壊れており修正が数行で済むため本 issue に含めた。

## テスト

導出ロジックは純関数なので `lib/runs/__tests__/` と同じ流儀でユニットテストを書く。

- ステージ導出: 規則 1〜5 の各分岐と優先順（PR `MERGED` が `liveStatus` に優先すること、PR なしの `handed_off` / `reviewing` が自己申告どおり `review` になること）
- 条件導出: `blocked` / PR `CONFLICTING` / マージされず閉じられた PR がそれぞれ blocked を立てること、stage と直交していること（PR `MERGED` + `blocked` の自己申告 → `blocked @ merged`。緑で塗り潰さない）
- レイアウト: key の重複したノードが別の座標に描かれること、`dependsOn` の重複が 1 本の辺に畳まれること、自己参照が辺にならないこと、循環時に最小列が 0 に寄ること
- レイヤリング: 依存なし / 直線チェーン / 複数親 / **循環**（無限再帰しないこと）
- 行割り当て: 親の行に子が寄ること、行が競合したら下に詰むこと

描画側は既存レンズと同様、スナップショット的な検証は行わない。

## やらないこと

- 進捗ファイルのスキーマ変更
- グラフライブラリの導入
- pan / zoom / ドラッグなどのインタラクション
- 交差最小化アルゴリズム
- run 一覧（run 間）の表現変更 — 今回は run 内のみ
