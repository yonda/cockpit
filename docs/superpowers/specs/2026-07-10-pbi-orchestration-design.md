# PBI オーケストレーション（エージェントチーム運用）設計

- 日付: 2026-07-10
- ステータス: 設計承認済み・実装前
- 前提: Launch Pad（`2026-07-09-launch-pad-design.md`）の完成が依存。本設計はその上位層

## ビジョン

**自分＝マネージャー、エージェント＝チームメンバー、cockpit＝インターフェースの中心。**

現状の運用（wezterm/herdr/agmsg でペインを覗き込みながら指示する）は、タスクごとの関与が深すぎて認知負荷とコンテキストスイッチが大きい。特に大きいペインは:

1. **糊作業**（最大のペイン）: 完了に気づく → 結果を確認 → 次を選ぶ → 指示文を書いて渡す、をタスク数×エージェント数ぶん手作業でやっている
2. **方向修正**: 要件解釈ミス・スコープ膨張による手戻り

スクラムの構図に載せて、人間の関与を 3 点に圧縮する:

| 関与ポイント | 内容 | 頻度 |
|---|---|---|
| スプリント計画 | Claude Code と対話して Definition of Ready を満たす PBI Issue を作る | PBI ごと（深い関与、ここは残したい仕事） |
| 分解承認 | エージェントのタスク分解案を承認/修正 | PBI ごとに 1 回〜 |
| PR レビュー & マージ | 全 PR を人間が見る。マージが「次へ進め」の合図 | 成果物ごと |

それ以外（タスク間の接続、進捗管理、レビューコメント往復）は自走する。

## ロードマップ（サブプロジェクト分解）

1. **Launch Pad 完成**（実施中）— 「1 Issue → 1 draft PR」の基礎部品
2. **PBI オーケストレーション層（本設計）** — PBI → 分解 → 承認 → タスク連続実行 → マージ検知で次へ
3. **レビューキュー（受信箱）** — 「要対応」を一列に並べる cockpit の新しい玄関（本設計に UI 方針を同梱）
4. **ガードレール & フォーマット整備 / Findy 展開** — PBI テンプレート、リポジトリ別設定、行動制限（本設計に方針を同梱）

## 決定事項（ブレインストーミングでの合意）

| 論点 | 決定 |
|---|---|
| PBI と PR の関係 | 1 PBI = 複数 PR。分解はエージェントが行う |
| 分解の承認 | 実行開始前に人間の承認ゲートを必ず挟む |
| 中間 PR のマージ | 全 PR を人間がレビュー & マージ（自動マージしない） |
| 「次へ進め」のトリガー | PR マージの検知のみ自動。それ以外の発射はすべて人間の明示操作 |
| レビューコメント対応 | 自動発射しない。受信箱からワンタップ発射 |
| 適用範囲 | 仕事リポジトリ（Findy）まで展開する。dogfood は cockpit 自身から |
| 実行環境 | ローカル Mac（Claude Code サブスクリプション内で完結させるため）。隔離環境は採らない |
| オーケストレータの形 | Runner 拡張型（決定論的状態機械）。リードエージェント型・GitHub Actions 駆動型は不採用 |
| 状態の置き場 | GitHub（親 Issue + sub-issues + PR）が唯一の真実。runner は最小限の進行状態のみ |

アプローチ選定の理由: 「知性が必要な仕事（分解・実装）はエージェントジョブに、進行管理は決定論的な機械に」という分離が、フォーマット・ガードレール・ワークフローの定義で楽をするという方針の実装形だから。リードエージェント型は「リードが今何を考えているか見に行く」というペインが一段上で再発するリスクがあり、GitHub Actions 駆動型は実行がどのみちローカル必須（サブスク制約）で二重インフラになるため不採用。

## PBI のライフサイクルと状態機械

```
[スプリント計画]                          （人間 + Claude Code、対話）
      │  Definition of Ready を満たす PBI Issue
      ▼
[発射] cockpit から PBI を launch          （人間、ワンタップ）
      ▼
[分解] エージェントジョブが PBI を読んでタスク分解案を作成
      │  → sub-issues として親 Issue にぶら下げる（proposed マーカー付き）
      ▼
[承認ゲート] cockpit に分解案が届く         （人間: 承認 / 修正指示 / 却下）
      ▼
[実行ループ] runner が決定論的に回す:
      依存が解けた sub-issue を選ぶ
      → Launch Pad ジョブ発射（worktree → 実装 → draft PR）
      → 人間が PR をレビュー & マージ
      → runner がマージ検知 → sub-issue クローズ → 次へ
      ▼
[完了] 全 sub-issue 完了 → 完了通知 → 動くプロダクトで受け入れ確認 → PBI クローズ
```

`PbiJob` の状態機械: `decomposing → awaiting_approval → executing → completed`（どの状態からも `failed` / `cancelled` へ。実行中の子ジョブ失敗やエスカレーションは `executing` 内の保留として表現）。

原則:

- **分解もただのエージェントジョブ**。Launch Pad の executor・許可転送を再利用し、成果物が「PR」ではなく「タスク分解の構造化出力」なだけ
- **実行ループは知性ゼロの機械**。「次のタスクを選ぶ → ジョブを撃つ → マージを待つ」だけ。判断に迷う状況はすべて受信箱へエスカレーション
- マージ検知はローカル runner からの GitHub ポーリング（1〜2 分間隔。webhook は使わない）

## PBI テンプレート（Definition of Ready）

スプリント計画セッションの成果物はこの形式の Issue（`.github/ISSUE_TEMPLATE/pbi.md`、`pbi` ラベル）:

- **ストーリー**: 何のために何を実現するか
- **受け入れ基準**: 完了を判定できる具体的な条件（箇条書き）
- **非スコープ**: やらないこと（方向ミスの最大の予防線）
- **技術的な手がかり**（任意）: 計画セッションで見えた設計方針・触るべきファイル・参考 PR

cockpit の発射台は `pbi` ラベル付き Issue を表示し、Definition of Ready を満たさないものにはワーニングを出す（発射は可能 — 形式チェックはガードレールであって手錠にしない）。

## 分解フェーズと承認ゲート

分解ジョブの構造化出力:

- タスク一覧（各タスク: タイトル / 目的 / 成果物 = PR の中身 / 受け入れ基準 / 依存タスク）
- 分解の根拠（1 PR = 1 revert 単位の原則でなぜこの切り方か）
- 不明点・リスク（承認時に人間が答えられる）

runner はこれを sub-issues として GitHub に作成（本文に proposed マーカー）し、`awaiting_approval` に遷移して受信箱に届ける。

承認ゲートの選択肢:

1. **承認** → sub-issues 確定、実行ループ開始
2. **修正指示** → コメントを添えて再分解ジョブへ（前回案 + 指摘を入力に再実行。往復は何度でも）
3. **却下** → PBI ジョブ中止（PBI 自体を書き直すべきケース)

再分解は sub-issues を削除・再作成せず、proposed のまま上書き更新する（GitHub 上のノイズ抑制）。

## 実行ループの詳細

**タスク選択と並行度**: sub-issues の依存グラフに従い、依存が解けたタスクから発射。同一 PBI 内でも独立タスクは並行可。並行度は runner の既存スケジューラの同時実行キャップを全 PBI で共有（現在 2、cockpit から変更可能にする）。

**ブランチと worktree**（既存規約に準拠）:

- タスクごとに `git wt`、ブランチは `feature/<sub-issue番号>-<短い説明>`
- ベースは必ず `git fetch origin` 後の `origin/<デフォルトブランチ>`。先行タスクの PR がマージ済みならその変更を自然に取り込んで始まる（逐次人間マージと噛み合う）
- PR は draft 限定（既存 allowlist）

**マージ検知と自動接続**（ポーリングで PR の状態変化を検知）:

- **マージ** → sub-issue クローズ → 依存が解けたタスクを発射（「指示し直し」の完全な代替）
- **マージなしクローズ** → 「このタスクをどうするか」を受信箱へ
- **レビューコメント** → 受信箱に通知。人間がワンタップで「コメント対応ジョブ」を発射 → エージェントが対応・push・スレッド返信（既存の reply-review-comments の流儀）

**PBI 完了**: 全 sub-issue クローズで `completed`。完了通知 → 動くプロダクトで受け入れ確認。不備が軽ければフォロータスクを PBI に追加（分解承認と同じ UI で承認）、重ければ新 PBI へ。

## 失敗・例外・エスカレーション

受信箱に届く「要対応」は 5 種類:

| 種類 | 発生源 | 操作 |
|---|---|---|
| 分解承認 | 分解ジョブ完了 | 承認 / 修正指示 / 却下 |
| 許可リクエスト | 実行中エージェント（既存の許可転送） | 許可 / 拒否 |
| 質問 | 実行中エージェントの判断迷い | 回答入力 |
| レビュー待ち PR | タスク PR 作成 / コメント対応 push | GitHub でレビュー → マージ or コメント → ワンタップ対応発射 |
| 失敗・異常 | ジョブ失敗、マージなしクローズ等 | リトライ / スキップ / PBI 中止 |

- **リトライ（指示付き）**: 失敗の経緯 + 人間の一言を添えて同タスクを再発射。worktree は作り直し
- **スキップ**: タスクを保留にし、依存しない他タスクは進める
- **PBI 中止**: 実行中の子ジョブをキャンセルし、未完了 sub-issues にコメントして停止
- **一時停止**（途中の仕様変更）: 実行中ジョブは完了まで走らせ、新規発射だけ止める。sub-issues を編集して再開
- **再起動耐性**: 起動時に sub-issues と PR の状態を GitHub と突き合わせて実行ループを再構築（Launch Pad の boot resume の拡張）
- **暴走ガード**: 1 タスクあたり実行時間・ターン数の上限。超過は失敗扱いでエスカレーション。自動リトライはしない（リトライは常に人間の判断）

## cockpit の UI

情報設計: **受信箱 = 動く場所、PBI ボード = 眺める場所、発射台 = 始める場所**。

- **受信箱（Inbox）**: 5 種類の要対応が新しい順に一列。各アイテムはインラインで完結（PR レビューのみ GitHub へリンク）。PWA 通知と連動し、iPhone からも許可・質問・分解承認に応答できる
- **PBI ボード**: 走っている PBI のカード（sub-issues 完了数 / 実行中タスク / 待ちバッジ）。詳細はタスク依存グラフ、ライブログ（Launch Pad 既存機能の流用）、一時停止・中止・フォロータスク追加
- **発射台**: 既存 `/launch` の発展。`pbi` ラベル付き Issue 一覧から発射
- 既存の PR/WIP ボードは「人間チームの PR も含めた自分の全景」として残す

## ガードレールと Findy 展開

**リポジトリ別設定**（cockpit に登録制。登録リポジトリのみ発射可能）: デフォルトブランチ / worktree 置き場（`../<repo>-wt/` 規約）/ 同時実行上限 / テスト・lint コマンド（DoD チェック用）/ 環境変数（`NX_DAEMON=false` 等）。

**行動制限**（Launch Pad 既存機構の継承・拡張）:

- PR 作成は draft 限定。`gh pr merge`・`gh pr ready`・デフォルトブランチへの直 push は禁止リスト
- push は自分のタスクブランチのみ、ファイルシステムは自分の worktree 内のみ
- 破壊的操作（force push、ブランチ削除等）は許可転送でエスカレーション

**Definition of Done ゲート**（PR 作成前の機械チェック）: リポジトリ設定のテスト・lint + ローカル `/code-review` セルフレビュー（Findy 既存規約）。通らなければ PR を作らず、自力修正かエスカレーション。人間のレビューは設計・仕様の妥当性に集中できる。

**展開順序**: cockpit 自身で dogfood（本設計自体を PBI として食わせる）→ 個人リポジトリ → Findy リポジトリ。

## テスト戦略

Launch Pad と同じ流儀。オーケストレータ（状態機械・タスク選択・マージ検知への反応）は GitHub クライアントと executor を injectable にして vitest で決定論的にテスト。分解ジョブの構造化出力はスキーマ検証。dogfood（実 PBI を 1 本流す）が結合テスト。

## 本設計のスコープ外

- 中間 PR の自動マージ（信頼データが溜まってから再検討）
- 隔離実行環境（コンテナ/クラウド）
- スプリント計画セッション自体の cockpit 内蔵（当面は従来どおり Claude Code の対話で行い、成果物の PBI Issue だけが cockpit に入る）
- 複数エージェント種別（codex 等）の混成チーム
